PROJECT 01 · June 2026

RFID Access Control System

A two-processor smart lock that reads, verifies and grants access in real time.

EmbeddedHardware

The problem & engineering challenge

A single microcontroller couldn't both poll an RFID reader over SPI and drive a lock, LCD, LEDs and buzzer without timing conflicts. The challenge was to split responsibilities across two processors and keep them in sync through a reliable communication link — while persisting authorized cards so the system survives a power cycle.

Key deliverables

  • Two-processor architecture: Arduino + MFRC522 reads UIDs over SPI; an AT89C51 (8051) drives the lock, LCD, LEDs and buzzer.
  • UID verification against records stored in EEPROM, with auto-enrollment of new cards.
  • Grant/deny decisions exchanged over a serial command link between the two controllers.
  • Full circuit designed and simulated in Proteus before hardware assembly.

PROJECT 02 · June 2026

AES-128 Encryption Simulator

A from-scratch AES implementation that you can actually watch encrypt, step by step.

SecurityAlgorithms

The problem & engineering challenge

AES is usually a black box hidden inside a library. I wanted to understand — and teach — exactly what happens in each round, which meant implementing the cipher from first principles with no crypto libraries, including arithmetic over the GF(2⁸) finite field, and then making the internal state legible through a live visualization.

Key deliverables

  • Full AES-128 built from scratch: SubBytes, ShiftRows, MixColumns over GF(2⁸), and AddRoundKey.
  • Complete key expansion generating all 11 round keys.
  • Tkinter GUI that renders the 4×4 state matrix after every step and round.
  • A key-expansion mode visualizing RotWord, SubWord, Rcon and the g-function.

Core

Python (no crypto libs)GF(2⁸) arithmeticS-Box / key schedule

Interface

Tkinter GUIState-matrix visualization

PROJECT 04 · December 2025

Cinema Booking System

Real-time seat selection that never double-books, backed by a transactional database.

Full-StackDatabase

The problem & engineering challenge

Seat booking is a classic concurrency problem: two users must never reserve the same seat. The challenge was modeling seat availability and booking transactions so the database stays consistent under real-time updates, while keeping the selection UI fast and intuitive enough that the flow feels effortless.

Key deliverables

  • Real-time seat selection and reservation management end to end.
  • Back-end logic and database integration managing seat availability, booking transactions and system updates.
  • An intuitive, user-friendly interface that streamlines the full booking journey.
